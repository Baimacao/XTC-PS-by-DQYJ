# 1. 提供应用基础信息

完善的应用信息是快速通过审核的必要前提，每个应用版本提交时，您需同时提供满足以下要求的材料，若信息无变更，可不必重复提交。

信息提供请参考“应用基础信息模板”

应用基础信息模板.xlsx

# 1.1 应用名

提供准确的应用名。

# 1.2 应用APK包

安装包大小不超过30M，安装后空间占用不超过40M。

# 1.3 应用图标

  • 尺寸：148*148px

  • 格式：PNG

  • 形状：直角方形

# 1.4 应用介绍图

  • 尺寸：320*360px

  • 格式：PNG

  • 形状：方形

  • 数量：3-5张

# 1.5 应用简介

20字以内描述。

# 1.6 应用详情

100字以内详细介绍，且与应用功能必须一致，禁止误导用户。

# 1.7 用户协议及隐私政策

需单独提供https格式的用户协议和隐私政策链接

# 1.8 应用软著

需提供应用软著文件，不限png/jpg格式

# 1.9 更新说明

请提供向用户展示的版本更新说明，另外，为帮助小天才测试，可同时提供详细的改动项清单。

# 1.10 开发者信息

如实提供开发者企业信息。

# 1.11 联系方式

用于用户咨询、问题反馈，付费应用要求必须提供服务号或服务热线。

# 1.12 应用使用说明

请提供详细的功能使用介绍，此介绍将同步至小天才客服部门，用于辅助解答用户咨询。

# 4. 版本提供

  • 新应用上架时请将应用基础信息、测试报告等 打包发送至developer@eebbk.com；迭代版本时，将详细改动说明、测试报告，打包发送至developer@eebbk.com.

  • 邮件格式要求

    标题：【版本验收】xx应用

    版本号：xx 版本名称: xxx

    正文：附上更新内容

# 5. 版本质量审核标准

  • 提测版本发现类似"某个模块不可用"，阻塞验收测试的严重必现问题，暂停验收，提测版本直接打回。

  • 应用初版发现基础功能中有5个及以上的必现问题；应用迭代版本发现基础功能中有3个及以上的必现问题，暂停验收，提测版本直接打回。

  • 首次打回，两个工作日后才会验收新版本；第二次打回，四个工作日后才会验收新版本；第三次打回，新版本验收需排期等待。

  • 版本打回以邮件通知方式告知，正常验收反馈以沟通为主，不在此范围内。

# 6. 其他注意事项

  • 每次给到的 APK versionCode 必须在上次的基础上+1。

  • 由于手表存在多个型号，所以可能会存在各个型号之间功能的差异化。但是最好使用一个 APK 兼容这些差异，可以通过读取系统型号（android.os.Build.MODEL）进行处理。

  • 每次给到的 APK packageName 不能变动。
